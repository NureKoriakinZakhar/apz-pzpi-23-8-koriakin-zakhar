3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про шаблони (патерни) проєктування
Шаблони (патерни) проєктування — це перевірені часом, стандартизовані архітектурні підходи до розв'язання поширених проблем проєктування об'єктно-орієнтованого програмного забезпечення. Вони не є готовими фрагментами коду, а слугують загальною концепцією вирішення певного класу задач. Базовим стандартом у цій галузі є класифікація «Банди чотирьох» (GoF), згідно з якою 23 класичні патерни поділяються на три групи: твірні (оптимізують створення об'єктів), структурні (визначають способи побудови складних систем) та поведінкові (регламентують алгоритми взаємодії). Головна область застосування патернів полягає у створенні гнучкої, масштабованої архітектури, яка легко піддається супроводу.


3.2 Опис обраного патерна
* Назва патерна: Адаптер (Adapter)
* Тип шаблона (патерна): структурний
* Основне призначення: організація спільної роботи об'єктів або підсистем, які мають несумісні інтерфейси. Шаблон діє як проміжний шар (обгортка), який трансформує виклики від клієнтського коду у формат, зрозумілий для стороннього або застарілого компонента.
* Передумови застосування: патерн доцільно використовувати при інтеграції сторонніх бібліотек з несумісним API; під час роботи із застарілим (Legacy) кодом, який неможливо модифікувати; а також для ізоляції бізнес-логіки від специфіки реалізації зовнішніх сервісів.
* Переваги: дотримання принципу єдиної відповідальності (SRP) шляхом ізоляції логіки конвертації; дотримання Принципу відкритості/закритості (OCP), що дозволяє додавати нові адаптери без зміни клієнтського коду.
* Недоліки: загальне ускладнення архітектури програмного продукту через необхідність введення додаткових абстракцій, інтерфейсів та класів-посередників.


3.3 Структура шаблона (патерна)
* Опис структурних частин:
   * Цільовий інтерфейс (Target): стандартний доменний інтерфейс, з яким очікує працювати клієнтський код.
   * Клієнт (Client): існуюча підсистема, що містить бізнес-логіку та взаємодіє з об'єктами через інтерфейс Target.
   * Адаптований клас (Adaptee): сторонній або застарілий клас, який містить корисний функціонал, але має несумісний з Клієнтом інтерфейс.
   * Адаптер (Adapter): клас-посередник, який реалізує інтерфейс Target і містить посилання на об'єкт Adaptee.
* Опис взаємодії об’єктів: Клієнт ініціює запит через метод Цільового інтерфейсу. Адаптер перехоплює цей виклик, за потреби конвертує параметри та делегує виконання специфічному методу всередині об'єкта Adaptee. Після виконання Адаптер отримує результат, адаптує його під вимоги Клієнта і повертає назад.
* UML-діаграми з поясненнями


  



На діаграмі зображено зв'язок реалізації (від Adapter до Target) та зв'язок композиції/агрегації (від Adapter до Adaptee).
3.4 Реалізація шаблона (патерна)
* Опис реалізації мовою програмування C#
* Основні етапи:
   * Створення базових інтерфейсів: розробка сучасного інтерфейсу ITarget, який оперує даними у форматі JSON, та залучення старого класу LegacySystem, що повертає дані у форматі XML.
   * Реалізація конкретних класів: створення класу XmlToJsonAdapter, який імплементує ITarget. Усередині методу він викликає LegacySystem, отримує XML і трансформує його у JSON.
   * Демонстрація використання: у методі Main створюється екземпляр LegacySystem, обгортається в XmlToJsonAdapter, після чого клієнтський код успішно отримує дані у сучасному форматі.

Приклади програмного коду та запити до ШІ
1  // 1. Сучасний Цільовий інтерфейс (Target)
2  public interface ITarget 
3  {
4      string GetEmployeeDataJson();
5  }
6  
7  // 2. Застаріла система (Adaptee)
8  public class LegacySystem 
9  {
10     public string GetXmlData() 
11     {
12         Console.WriteLine("LegacySystem: Звернення до старої БД...");
13         return "<employee><name>John Doe</name><role>Developer</role></employee>";
14     }
15 }
16 
17 // 3. Адаптер (Adapter)
18 public class XmlToJsonAdapter : ITarget 
19 {
20     private readonly LegacySystem _legacySystem;
21 
22     public XmlToJsonAdapter(LegacySystem legacySystem) 
23     {
24         _legacySystem = legacySystem;
25     }
26 
27     public string GetEmployeeDataJson() 
28     {
29         string xmlData = _legacySystem.GetXmlData();
30         Console.WriteLine("Adapter: Конвертація XML у JSON...");
31         // Імітація конвертації
32         string jsonData = "{ \"employee\": { \"name\": \"John Doe\", \"role\": \"Developer\" } }";
33         return jsonData;
34     }
35 }
36 
37 // 4. Клієнтський код (Client)
38 class Program 
39 {
40     static void Main() 
41     {
42         LegacySystem oldSystem = new LegacySystem();
43         ITarget adapter = new XmlToJsonAdapter(oldSystem);
44 
45         string result = adapter.GetEmployeeDataJson();
46         Console.WriteLine($"Client отримав: {result}");
47     }
48 }
